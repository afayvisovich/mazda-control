# Анализ протокола управления автомобилем через ADB Logcat

## Краткое резюме (Executive Summary)

**Цель:** Reverse-engineer бинарный протокол управления функциями автомобиля для реализации голосового интерфейса.

**Ключевые открытия:**
- ✅ **Протокол:** 30-байтный заголовок + RealBody, команды через localhost:32960
- ✅ **Багажник:** Property ID `0x66000210`, RealBody<41>=`07` (open)/`06` (close), ~8.6 сек операция
- ✅ **Спойлер:** Property ID `0x6600022C`, RealBody<41>=`06` (open)/`05` (close), ~2.2 сек движение
- ✅ **Окна — 5 режимов управления** (детали ниже):
  - **Индивидуальное водительское:** RealBody<41>=`03`, Function<43>=`24`/`25`/`27`
  - **Индивидуальное пассажирское:** RealBody<41>=`01`, Function<43>=`27`/`28`
  - **Все окна (групповое):** RealBody<41>=`01`, Function<43>=`29`/`25`
  - **Два передних (групповое):** RealBody<41>=`04`, Function<43>=`1C`/`1B`/`1A`
  - **Задние окна (групповое):** RealBody<41>=`04`, Function<43>=`1B`/`1A`
- ✅ **Климат ON/OFF:** Property ID `0x66000023`, AC value: `1`=ON/`0`=OFF, ответ ~0.2-0.3 сек
- ✅ **Климат доп.:** Property ID `0x66000011` (вентилятор), `0x66000017` (режим), `0x6600000E` (качество воздуха)
- ✅ **Time<29> коды:** `02`=OPEN/ON, `0C`=CLOSE/OFF (консистентно для всех компонентов)
- ✅ **raw_value=48.0:** появляется в статусах HVAC/спойлера (возможно напряжение/ток)
- ✅ **Function Bytes (38-45) декодированы:**
  - **Byte 41 — Component ID:** `01`=FR, `03`=FL, `04`=Dual, `06`=Spoiler, `07`=Trunk, `12`=HVAC
  - **Byte 43 — Action Code:** `24`=50% FL, `25`=Close, `27`=100% open, `28`=100% FR, `29`=All open, `1A/1B/1C`=Dual windows

**⚠️ Критические расхождения с предыдущей версией документации:**
- **ОШИБКА:** Указано "RealBody<41>=`03` для передних окон" — **НЕВЕРНО**
  - **ИСПРАВЛЕНО:** RealBody<41>=`03` ТОЛЬКО для индивидуального водительского окна
  - RealBody<41>=`01` для индивидуального пассажирского окна
  - RealBody<41>=`04` для группового управления двумя передними окнами
- **ОШИБКА:** Описано только 2 режима управления окнами (индивидуальное vs групповое) — **НЕВЕРНО**
  - **ИСПРАВЛЕНО:** Выявлено **5 различных режимов управления** с уникальными комбинациями RealBody/Function
- **УПУЩЕНО:** Функциональные коды `1A`/`1B`/`1C`/`29` полностью отсутствовали в документации — **ДОБАВЛЕНЫ**

**🔄 Обновления от марта 2026 (тестирование в автомобиле):**
- **Counter (Byte 355):** Уменьшается на 1 каждую секунду при polling (0xB7 → 0xB6)
- **Command Mark <363>:** `0x5A` = команда (исходящая), `0x0A` = ответ от сервера (входящий)
- **Time Byte 29:** `0x02` = OPEN/ON, `0x0C` = CLOSE/OFF, `0x16` = 50% позиция
- **Процессы автомобиля:** `com.mega.car` (PID 6624), `com.mega.controller` (PID 7414)
- **Порт подключения:** `localhost:32960`

**Готово к реализации:** Структура пакетов, Property IDs, коды команд для багажника, спойлера, всех окон, климата ON/OFF.

**Требует исследования:** Управление скоростью вентилятора, температурой, режимами обдува, алгоритм контрольных сумм.

## Архитектура системы управления

### Компоненты:

1. **Головное устройство (Head Unit)** - Android 11
2. **Локальный сервер команд** - `localhost:32960`
3. **Сервисы автомобиля:**
   - `CarPropertyService` - управление свойствами автомобиля
   - `CanSignalHandler` - обработка CAN-сигналов
   - `HvacService` - управление климатом
   - `CarPropManager` - менеджер свойств

### Протокол обмена:

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Пакет команды (512 байт)                     │
├──────────────┬──────────────────────────────────────────────────────┤
│ Header (30B) │ 23 23 XX FE 4C 56 52 48 44 41 45 4A ...            │
│ Encrpy-mark  │ 01 (флаг шифрования)                                │
│ Length (2B)  │ XX XX (длина тела)                                  │
│ Time (6B)    │ XX XX XX XX XX XX (временная метка YY MM DD HH MM SS)│
│ RealBody     │ [Основные данные команды]                           │
└──────────────┴──────────────────────────────────────────────────────┘

Структура пакета:
  0     1     2     3     4           20    21    22    23    24          29    30
┌─────┬─────┬─────┬─────┬────────────┬─────┬─────┬─────┬─────┬───────────┬─────┐
│ 23  │ 23  │ Mark│ FE  │ VIN (17B)  │Enc  │ Len │ Len │ Time (6B)     │Body │ ...
└─────┴─────┴─────┴─────┴────────────┴─────┴─────┴─────┴─────────────────┴─────┘
         Command                      (01)  (Hi) (Lo)  YY MM DD HH MM SS
         Mark
```

## Типы команд (Command Mark)

| Mark | Назначение | Пример использования |
|------|-----------|---------------------|
| `2`  | Команда управления (действие) | Открытие/закрытие багажника, спойлера, окон |
| `3`  | Запрос статуса (polling) | Опрос текущего состояния, подтверждение выполнения |

### Детализация Command Mark:

**Command Mark = 2 (Исходящая команда):**
- Отправляется приложением к серверу
- Byte 363 = `0x5A`
- Time Byte 29 кодирует действие:
  - `0x02` = OPEN/ON
  - `0x0C` = CLOSE/OFF
  - `0x16` = 50% позиция (окна)

**Command Mark = 3 (Входящий ответ/Status Polling):**
- Ответ от сервера приложению
- Byte 363 = `0x0A`
- Counter (Byte 355) уменьшается на 1 каждую секунду
- Пример: `0xB7` → `0xB6` → `0xB5`

## Управление багажником (Trunk)

### Property ID:
- **Decimal:** `1711276336`
- **Hex:** `0x66000210`
- **Сигнал:** `VEHICLEDOOR_BCM_TRUNKDOORSTATUS`

### Значения статуса:
| Значение | Состояние |
|---------|-----------|
| `0` | Закрыт |
| `1` | Открыт |

### Дополнительный параметр позиции:
- **Сигнал:** `VEHICLEDOOR_DOOR_PTS_GATEPOSITION`
- **Диапазон:** `0.0` - `100.0` (проценты открытия)

### Логика работы:

**boot.txt** (открытие → ожидание 5 сек → закрытие):
```
22:10:11.532 - VEHICLEDOOR_BCM_TRUNKDOORSTATUS = 1 (открыт)
22:10:11.540 - TTS: "The Boot Lid Is Open"
22:10:12.892 - command mark = 2 (команда на закрытие)
```

**boot2.txt** (открытие → без закрытия):
```
22:10:33.637 - command mark = 3 (только опрос статуса)
22:10:33.641 - upload stick : fail (нет команды закрытия)
```

### Структура команды управления:

**Команда открытия/закрытия (command mark = 2):**
```
Header:
  <0-2>:  23 23 02 (mark = 2)
  <3>:    FE
  <4-20>: 4C 56 52 48 44 41 45 4A 38 52 4E 30 31 33 35 37 39
Encrpy-mark: <21>: 01
Length: <22-23>: 01 9C (412 байт)
Time: <24-29>: 1A XX XX XX XX XX

RealBody (ключевые байты):
  <30-33>: 01 02 03 01 (префикс команды)
  <34-35>: 00 00 (параметры: открытие/закрытие)
  <38-45>: BF 11 0D XX 27 XX 20 01 (идентификатор функции)
  ...
  <355>: 2B (контрольная сумма/тип)
  <363>: 5A (маркер конца)
```

### Отличия команд:

| Параметр | Открытие | Закрытие |
|---------|---------|----------|
| Time байт <29> | `02` | `0C` |
| RealBody <41> | `07` | `06` |
| RealBody <365> | `00` | `10` |

### Property ID:
- **Decimal:** `1711276588`
- **Hex:** `0x6600022C`
- **Сигнал:** `VEHICLEDOOR_DOOR_PTS_SPOILERPOSITION`
- **Диапазон:** `0.0` - `100.0` (проценты, в логах 0-101)

### Логика работы (spoiler.txt):

**Полный цикл: Открытие → Закрытие → Открытие**

```

22:11:55.430 - sendControlMsg: {"type":"change_view","params":{"view_id":8}}
22:11:55.734 - command mark = 3 (опрос статуса)
```

### Команды:
- **Поднятие:** установка позиции в `100.0`
- **Опускание:** установка позиции в `0.0`
- **Стоп:** фиксация текущей позиции

### Отличия команд:

| Параметр | Открытие | Закрытие |
|---------|---------|----------|
| Time байт <29> | `02` | `0C` |
| RealBody <41> | `06` | `05` |
| RealBody <365> | `00` | `00` |

**Полные байты функции (RealBody <38-45>):**
- **OPEN:** `BF 11 0D 06 27 21 20 01`
- **CLOSE:** `BF 11 0D 05 27 23 20 01`

### Система обратной связи по позиции:

Спойлер предоставляет непрерывную обратную связь во время движения:

| mMovement | Состояние | Описание |
|-----------|-----------|----------|
| `1` | Стационарно | Движение завершено, позиция стабильна |
| `2` | Открытие | Движение вверх (0% → 100%) |
| `3` | Закрытие | Движение вниз (100% → 0%) |

**Частота обновления:** ~100 мс во время движения

### Сравнение с багажником:

| Параметр | Багажник | Спойлер |
|---------|---------|---------|
| Command Mark | 02 | 02 |
| Property ID (Hex) | `0x66000210` | `0x6600022C` |
| Property ID байты | `00 00 00 00` | `00 00 00 00` |
| Open action (RealBody<41>) | `07` | `06` |
| Close action (RealBody<41>) | `06` | `05` |
| Время срабатывания | ~1.5 сек | ~2.0 сек |

**Вывод:** Спойлер использует ту же структуру протокола, что и багажник, но с другими кодами действий и Property ID.

## Декодирование Function Bytes (байты 38-45)

### Структура Function Bytes:

```
Байты:  38    39    40    41    42    43    44    45
        ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
Значение: BF    11    0D   [ID]   27   [Act]  20    01   │
        └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
         Префикс       Комп  Констан  Дейст  Суффикс
         (3 байта)     онент   0x27    ие     (2 байта)
```

### Поля Function Bytes:

| Байт | Название | Значение | Описание |
|------|----------|----------|----------|
| 38-40 | Префикс | `BF 11 0D` | Константный префикс для команд |
| **41** | **Component ID** | `01`/`03`/`04`/`06`/`07`/`12` | **Идентификатор компонента** |
| 42 | Константа | `27` | Фиксированное значение |
| **43** | **Action Code** | `1A`-`29` | **Код действия** |
| 44-45 | Суффикс | `20 01` | Константный суффикс |

### Component ID (Byte 41):

| Значение | Компонент | Property ID (Hex) | Файлы |
|----------|-----------|-------------------|-------|
| `0x01` | Переднее правое окно (FR) | `0x6600010E` | `pas_wind_*.txt` |
| `0x03` | Переднее левое окно (FL) | `0x6600010C` | `dr_wind_*.txt` |
| `0x04` | Два окна (Dual) | - | `b_dr_pas_wind_*.txt`, `b_pas_wind_*.txt` |
| `0x06` | Спойлер | `0x6600022C` | `spoiler.txt` |

### Action Code (Byte 43):

| Значение | Действие | Компонент | Примечание |
|----------|----------|-----------|------------|
| `0x24` | 50% OPEN | FL окно | Начальное открытие |

| `0x26` | CLOSE | Задние окна | Групповое закрытие |
| `0x27` | 100% OPEN | FL/FR окна | Полное открытие |

### Response Packets (Command Mark = 3):

В ответных пакетах структура Function Bytes отличается:

```
Command packets (mark=2):  BF 11 0D [comp] 27 [action] 20 01
Response packets (mark=3): [timer] [timer] [comp] [varies] [action] [varies] [varies]
```

**Особенности response packets:**
- Байты 38-39: Таймер/счётчик (инкрементируется со временем)
- Байт 41: Component ID (аналогично командам)
- Байт 43: Action Code (может отличаться от команды)
- Значения зависят от позиции окна в момент опроса

## Управление климатом (HVAC)

### Property IDs:
- **0x66000023** (1711276067) - `AC/Status/Front/Set` - Вкл/Выкл климата
  - Значение: `1` = ВКЛ, `0` = ВЫКЛ
- **0x66000011** (1711276049) - `AC/WindLevel/Front` - Уровень вентилятора (статус)
  - Значение: `0` = выкл, `1+` = скорость (обновляется автоматически)
- **0x66000017** (1711276055) - `AC/ACMode` - Режим кондиционера (статус)
  - Значение: `0` = выкл, `1` = вкл
- **0x6600000E** (1711276046) - `SENSOR_INSIDEAIRQUALITY` - Качество воздуха в салоне
  - Значение: 7-10 (обновляется каждую секунду, наблюдается в логах)

### Команды управления (command mark = 2):

**ВЫКЛЮЧЕНИЕ климата:**
```
Время: 22:18:13.349, AC/Status/Front/Set value=0
Byte 34: 0x0D (команда выключения)
Byte 355: 0x7D (125)
Byte 363: 0x0A (10)
Byte 365: 0x00 (0)
```

**ВКЛЮЧЕНИЕ климата:**
```
Время: 22:18:15.031, AC/Status/Front/Set value=1
Byte 34: 0x12 (команда включения)
Byte 355: 0x7D (125)
Byte 363: 0x0A (10)
Byte 365: 0x00 (0)
```

### Временная шкала переключений (cl_off_on_off_on.txt):
```
22:18:13.349 - OFF (value=0, byte34=0x0D)
22:18:15.031 - ON  (value=1, byte34=0x12)  [+1.7 сек]
22:18:17.503 - OFF (value=0, byte34=0x0D)  [+2.5 сек]
22:18:19.105 - ON  (value=1, byte34=0x12)  [+1.6 сек]
```
**Всего циклов:** 4 переключения AC (OFF→ON→OFF→ON)

### Характеристики HVAC:
- **Время отклика:** ~0.2-0.3 сек на подтверждение статуса
- **Тип управления:** Дискретные состояния (0=OFF, 1=ON)
- **Обратная связь:** Нет позиции (в отличие от багажника/спойлера/окон)
- **Команда mark=2:** Отправка команды управления
- **Команда mark=3:** Опрос статуса/подтверждение
- **raw_value=48.0:** Появляется в статусах (аналогично спойлеру, возможно напряжение/ток)

### Логика работы:
```
HvacService обрабатывает команды через CarPropertyManager
При получении команды отправляется пакет с command mark = 2
Ответ приходит с command mark = 3 (подтверждение)
AC/WindLevel и AC/ACMode обновляются автоматически после команды
```

### Алгоритм формирования команды:
1. Property ID: 0x66000023 (AC/Status/Front/Set)
2. Команда ON: byte[34] = 0x12, value = 0x01
3. Команда OFF: byte[34] = 0x0D, value = 0x00
4. Bytes 355, 363, 365 остаются константами (0x7D, 0x0A, 0x00)

### ⚠️ Примечание:
- Управление скоростью вентилятора требует дополнительного исследования
- В тестах уровень вентилятора был установлен на 1 и не менялся
- При выключении климата вентилятор автоматически останавливается (value=0)
- Датчик качества воздуха показывает значения 7-10, обновляется каждую секунду

### 2. Формирование пакета команды

```java
public byte[] buildCommandPacket(int commandMark, int functionId, byte[] params) {
    ByteArrayOutputStream packet = new ByteArrayOutputStream();
    
    // Header
    packet.write(new byte[]{0x23, 0x23, (byte)commandMark, (byte)0xFE});
    packet.write(VIN_BYTES); // 4C 56 52 48 44 41 45 4A 38 52 4E 30 31 33 35 37 39
    packet.write((byte)0x01); // Encrpy-mark
    
    // Length
    int bodyLength = params.length + 20;
    packet.write((bodyLength >> 8) & 0xFF);
    packet.write(bodyLength & 0xFF);
    
    // Time (текущее время в формате YY MM DD HH MM SS)
    packet.write(getTimestampBytes());
    
    // RealBody
    packet.write(params);
    
    return packet.toByteArray();
}
```

### 3. Пример команды для багажника

```java

    // ... остальные байты
};

```

### 4. Мониторинг статуса

```java

// Чтение ответа

```

### 5. Обработка ответов

```java
public int parseStatus(byte[] response) {
    // Проверка заголовка
    if (response[0] != 0x23 || response[1] != 0x23) {
        throw new InvalidResponseException();
    }
    
    // Извлечение Command Mark
    int mark = response[2] & 0xFF;
    
    // Извлечение RealBody
    int bodyOffset = 30; // Header + Encrpy + Length + Time
    byte[] body = Arrays.copyOfRange(response, bodyOffset, response.length);
    
    // Парсинг статуса из body[34-35]
    return (body[34] & 0xFF) | ((body[35] & 0xFF) << 8);
}
```

## Критическое открытие: Двухуровневая архитектура Property ID

### 🔍 Проблема несоответствия Property ID

В ходе исследования обнаружено фундаментальное расхождение:
- **Документированные Android Automotive Property IDs**: `0x6600010C`, `0x66000210`, `0x6600022C` и т.д.
- **Наблюдаемые в пакетах Property IDs**: ВСЕГДА `0x00000000`

**Это НЕ ошибка кодирования — это особенность архитектуры!**

### 🏗️ Двухуровневая система Property ID

```
┌─────────────────────────────────────────────────────────────────┐
│                    Android Framework Level                      │
│  Property IDs: 0x6600010C, 0x66000210, 0x6600022C, ...         │
│  (документированные Android Automotive Property IDs)            │
└────────────────────┬────────────────────────────────────────────┘
                     │ VHAL (Vehicle HAL) Layer
                     │ Маппинг между уровнями
                     ▼
┌─────────────────────────────────────────────────────────────────┐
│                      VHAL Layer (протокол)                      │
│  Property ID в пакетах: ВСЕГДА 0x00000000                       │
│  Идентификация компонента: Function Byte[3]                     │
│  - 0x03 = Переднее левое окно (соответствует 0x6600010C)        │
│  - 0x01 = Переднее правое окно (соответствует 0x6600010E)       │
│  - 0x07 = Багажник (соответствует 0x66000210)                   │
│  - 0x06 = Спойлер (соответствует 0x6600022C)                    │
│  - 0x12 = HVAC (соответствует 0x66000023)                       │
└─────────────────────────────────────────────────────────────────┘
```

### 📊 Таблица маппинга Property ID

| Компонент | Function Byte[3] | Android Framework Property ID | Property ID в пакете |
|-----------|------------------|------------------------------|---------------------|
| Переднее левое окно (FL) | `0x03` | `0x6600010C` (1711276300) | `0x00000000` |
| Переднее правое окно (FR) | `0x01` | `0x6600010E` (1711276302) | `0x00000000` |
| Заднее левое окно (RL) | `0x02` | `0x6600010D` (1711276301) | `0x00000000` |
| Заднее правое окно (RR) | `0x04` | `0x6600010F` (1711276303) | `0x00000000` |
| Багажник (Trunk) | `0x07` | `0x66000210` (1711276336) | `0x00000000` |
| Спойлер (Spoiler) | `0x06` | `0x6600022C` (1711276588) | `0x00000000` |
| Климат (HVAC ON/OFF) | `0x12` | `0x66000023` (1711276067) | `0x00000000` |
| Климат (Fan Speed) | `0x12` | `0x66000011` (1711276049) | `0x00000000` |
| Климат (Mode) | `0x12` | `0x66000017` (1711276055) | `0x00000000` |

### 🔬 Подтверждение из логов

**Файл:** `dr_wind_50op_100op_50cl_100cl.txt`

```
22:16:47.271 CarPropertyService: onPropertyEvent: propertyId=0x6600010C, area=0, value=0.0
22:16:47.271 CarPropertyService: onPropertyEvent: propertyId=0x6600010C, area=0, value=50.0
22:16:47.271 CarPropertyService: onPropertyEvent: propertyId=0x6600010C, area=0, value=100.0
```

**Одновременно в пакете:**
```
Bytes 34-37 (Property ID): 00 00 00 00
Byte 38 (Function Byte[3]): 03
```

**Вывод:** `CarPropertyService` получает `0x6600010C`, но в протоколе передаётся `0x00000000` с Function Byte `0x03`.

### 📝 Как это работает

1. **Android Framework** отправляет команду с Property ID `0x6600010C` (водительское окно)
2. **VHAL (Vehicle HAL)** получает команду и маппит на внутренний формат:
   - Property ID → `0x00000000` (универсальный)
   - Function Byte[3] → `0x03` (идентификатор компонента)
3. **Протокол** передаёт пакет с `0x00000000` + Function Byte `0x03`
4. **Головное устройство** получает пакет, VHAL маппит обратно:
   - `0x00000000` + Function Byte `0x03` → `0x6600010C`
5. **CarPropertyService** обрабатывает команду для `0x6600010C`

### 💡 Почему это важно

**Для разработчика стороннего приложения:**

✅ **Хорошая новость:** Property ID в пакетах ВСЕГДА `0x00000000` — это нормально!

✅ **Идентификация компонента** происходит через Function Byte[3] (байт 38 в RealBody)

✅ **Не нужно беспокоиться** о соответствии documented Property IDs

⚠️ **Важно:** Function Byte[3] определяет компонент:
- `0x03` = водительское окно
- `0x01` = пассажирское окно
- `0x07` = багажник
- и т.д.
